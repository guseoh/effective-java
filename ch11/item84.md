## ✅ 프로그래밍의 동작을 스레드 스케줄러에 기대지 말라

## 1. 스레드 스케줄러

여러 스레드가 실행 중이면 운영체제의 스레드 스케줄러가 어떤 스레드를 얼마나 오래 실행할지 정한다.

정상적인 운영체제라면 이 작업을 공정하게 수행하지만 구체적인 스케줄링 정책은 운영체제마다 다를 수 있다.

프로그램이 스레드 스케줄러에 의존적이라면, 해당 프로그램의 동작이나 성능이 운영체제마다 다르게 나타날 수 있다. 이러한 프로그램은 다른 플랫폼에 이식하기 어렵다.


<br>

## 2. 프로그램의 성능과 이식성이 높은 프로그램 작성하는 방법

### 실행 가능한 스레드의 평균적인 수 >>> 프로세서 수 → X
- 멀티 스레딩 환경에서 스레드는 프로세서에서 실행된다.
- 만약 실행 가능한 스레드의 수가 프로세서 수보다 많으면, 각 스레드는 자주 실행 중단되고 다른 스레드가 실행되기 시작한다.
- 따라서 **문맥 교환과 스레드 간 경합으로 인해 프로세서의 자원 낭비와 성능 저하가 발생**할 수 있다.
- 여기서 실행 가능한 스레드의 수와 전체 스레드 수는 구분해야 한다.
    - 전체 스레드 수는 훨씬 많을 수 있고, 대기 중인 스레드는 실행 가능하지 않다.

<br>

### 실행 가능한 스레드 수를 적게 유지하는 방법
- **각 스레드가 작업을 완료한 후에는 다음 일거리가 생길 때까지 대기하도록 하는 것**
    - 스레드는 당장 처리해야 할 작업이 없다면 실행돼서는 안 된다.
    - 작업이 없는 스레드가 계속 실행 상태로 있으면, 불필요한 문맥 교환과 CPU 자원의 낭비가 심해지게 된다.
- ex) 실행자 프레임워크
    - 스레드 풀 크기를 적절히 설정하고 작업은 짧게 유지하면 된다.
    - 단, 너무 짧으면 작업을 분배하는 부담이 오히려 성능을 떨어뜨릴 수도 있다.

<br>

### 스레드는 절대 바쁜 대기(busy waiting) 상태가 되면 안 된다.
- 공유 객체의 상태가 바뀔 때까지 쉬지 않고 검사해서는 안 된다는 뜻이다.
<details>
    <summary>바쁜 대기?</summary>
<div markdown="1">

- **스레드가 특정 조건이 충족될 때까지 계속해서 해당 조건을 검사하면서 대기하는 것**을 말한다.
- 바쁜 대기 중인 스레드는 CPU 시간을 많이 차지하게 된다.
    - 따라서 다른 스레들이 실행될 기회가 줄어들게 되고, 전체 시스템의 성능이 저하될 수 있다.
- 바쁜 대기 상태의 스레드는 스레드 스케줄러의 동작을 예측하기 어려워 원하는 조건의 충족 시점을 놓칠 수 있으며, 이는 전체적인 성능과 동작의 예측성을 저하시킬 수 있다.

```java
while (!condition) {
    // 아무 작업도 안 하고 계속 확인
}
```

- 여기서 `condition`이 true가 될 때까지 스레드는 계속 루프를 돌며 CPU를 사용한다.
- CPU 점유율은 높아지고, 다른 스레드나 프로세스가 실행될 CPU 시간을 뺏기 때문에 비효율적이다.

</div>
</details>

<details>
    <summary>끔직한 CountDownLatch 구현 - 바쁜 대기 버전!</summary>
<div markdown="1">

```java
public class SlowCountDownLatch {
    private int count;
    public SlowCountDownLatch(int count) {
        if (count < 0)
            throw new IllegalArgumentException(count + " < 0");
        this.count = count;
    }
    // 바쁜 대기 구조
    public void await() {
        while (true) { 
            synchronized(this) {
                if (count == 0)
                    return;
            }
        }
    }
    public synchronized void countDown() {
        if (count != 0)
            count--;
    }
}
```
</div>
</details>

- 스레드가 `await()`를 호출하면 `count`가 0이 될 때까지 계속 루프를 돌면서 상태를 확인한다.
- `count`가 아직 0이 아니면 **CPU를 계속 점유한 채로 반복 확인** → `바쁜 대기`

## 3. 주의할 점

### Thread.yield을 사용해서 문제를 해결하려 하지 말자.

- `Thread.yield()` 은 현재 실행 중인 스레드가 스스로 “**나는 다른 스레드에게 CPU 기회를 양보할게**” 라고 힌트를 주는 메서드이다.
    - **CPU 점유를 완전히 포기하지 않고**, OS 스케줄러가 스케줄링을 결정할 뿐이다.
- 왜 문제 해결이 되지 않을가?
    1. `yield()` 를 호출해도 OS가 실제로 다른 스레드를 실행할지, 현재 스레드를 계속 실행할지는 보장되지 않는다.
    2. 루프를 계속 돌면서 조건을 확인하므로 성능 저하가 발생한다.
    3. `yield()` 는 단순히 스케줄링 힌트 정도로, 바쁜 대기 문제를 근본적으로 해결하지 못한다.

<br>

### 스레드 우선순위 조절하려고 하지 말자.
- 스레드 우선순위는 자바에서 이식성이 가장 나쁜 특성에 속한다.
- 스레드 안전순위 조정을 통한 성능 향상은 특정 환경에서만 작동할 수 있고, 다양한 환경에서의 안정적인 성능을 보장하지 못한다.


<br>

## 4. 핵심 정리
- 프로그램의 동작을 스레드 스케줄러에 기대하지 말자.
    - 견고성과 이식성을 모두 해치는 행위다.
- Thread.yield와 스레드 우선순위에 의존해서도 안 된다.
    - 이 기능들은 스레드 스케줄러에 제공하는 힌트일 뿐이다.
